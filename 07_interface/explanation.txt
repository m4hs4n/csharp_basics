An Interface is a strict contract that tells a class what it must do, but completely leaves it up to the class to decide how to do it.
Think of it like a remote control layout: it has a power button, a volume up button, and a volume down button.
Any TV manufacturer can build a TV to work with that remote, but a Sony TV might turn on differently behind the scenes than a Samsung TV.
In code, an interface contains only declarations (method names and return types) but no actual code inside them.

#Why do we use them?
 1 - Decoupling (Flexibility): It allows you to swap out different implementations easily without breaking the rest of your application.
 2- Standardization: It forces different developers on a team to use the exact same method names for similar tasks.


#The Real-World Example: Notification Systems
Imagine you are building a feature for a software that needs to send alerts to employees. Some employees want an Email, and others want an SMS.
Instead of writing separate, messy code for each type, we create an interface contract called INotificationService.

1. Define the Contract (The Interface)
By convention, all interface names in C# start with a capital I. Note that the method has no body { }, just a semicolon.

public interface INotificationService
{
    // Every class that signs this contract MUST have a Send method
    void Send(string message);
}

2. Implement the Contract (The Classes)
Now, we create classes that "sign" the interface contract using a colon : symbol. They are forced to fill in the code for the Send method.

// Implementation 1: Email
public class EmailNotification : INotificationService
{
    public void Send(string message)
    {
        Console.WriteLine($"Sending Email with content: {message}");
        // Actual email server logic would go here
    }
}

// Implementation 2: SMS
public class SmsNotification : INotificationService
{
    public void Send(string message)
    {
        Console.WriteLine($"Sending SMS text message: {message}");
        // Actual telecom API logic would go here
    }
}

3. Use the Interface Polymorphically (The Power of Interfaces)
Because both classes implement INotificationService, you can treat them interchangeably in your main program. Look how clean this is:

class Program
{
    static void Main(string[] args)
    {
        // We can store different classes inside the same interface type variable!
        INotificationService notification;

        // If the user prefers Email:
        notification = new EmailNotification();
        notification.Send("Your shift starts at 9 AM."); // Prints Email output

        // If the user switches preferences to SMS:
        notification = new SmsNotification();
        notification.Send("Your shift starts at 9 AM."); // Prints SMS output
    }
}

#Summary Cheat Sheet
-An interface has no behavior: It only lists method signatures (e.g., void Send(string msg);).
-Classes use : to implement it: They must provide the concrete code for every single method listed in the interface, or the compiler will throw an error.
-Interfaces achieve loose coupling: Your main code doesn't need to know if it's talking to an Email or an SMS class; it only cares that it can call .Send().
